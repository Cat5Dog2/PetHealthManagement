## 1. システム概要

### 1.1 システム名
- ペット健康管理アプリ（仮称）

### 1.2 システム種別

- Webアプリケーション
  - バックエンド：.NET 10 (LTS) / ASP.NET Core MVC
  - 言語：C# 14
  - ORM：Entity Framework Core 10
  - 認証・認可：ASP.NET Core Identity

### 1.3 対象プラットフォーム・インフラ
- 開発環境
  - OS：Windows
  - DB：SQL Server LocalDB
  - 実行：ローカル（Kestrel）
  - 機密情報管理：`dotnet user-secrets` 等
- 本番環境（想定）
  - アプリホスト：Azure App Service
  - DB：Azure SQL Database
  - 通信：HTTPS（TLS）
  - 機密情報管理：App Service の「構成」または Azure Key Vault

### 1.4 対象ユーザー・対象ペット
- 対象ユーザー：ペットを飼う一般ユーザー、開発者本人（管理者ユーザー）
- 対象ペット：犬猫を含む複数種（後述の Species 10分類）

### 1.5 システムの目的
- 一般ユーザーが複数のペットについて、以下の情報を一元管理できるようにする。
  - プロフィール情報（基本情報・画像）
  - 健康ログ（日々の体調・体重・食事・活動量など）
  - 予防・投薬などの予定（スケジュール）
  - 通院履歴（診断内容・処方など）
- ペットのプロフィール情報は、公開設定（IsPublic）に応じて **ログイン済みユーザー同士**で共有できるようにする（未ログインへの公開は行わない）。
- ペットの詳細な健康情報（健康ログ・予定・通院履歴）は、登録オーナーのみ閲覧・編集可能とし、「プロフィールの共有」と「健康情報のプライバシー」を両立する。
- 健康ログ・通院履歴には複数の画像を添付できるようにし、診断書や検査結果などを視覚的に残せるようにする。
- 一般ユーザー自身がプロフィール更新やアカウント削除を行えるようにし、自己完結したアカウント管理を可能とする。
- 管理者機能（ユーザー一覧・削除）を用意するが、管理者ロールによって他ユーザーのペット/健康情報の閲覧権限は増やさない。

## 2. 利用者とロール

### 2.1 ロール定義
- 一般ユーザー  
  メールアドレスとパスワードで登録・ログインして利用するペットオーナー。自身のペットとその健康情報を管理する。

- 管理者ユーザー  
  開発者本人を想定した特別ユーザー。ASP.NET Core Identity のロール「Admin」を保持する。  
  「Admin」ロールは DB を直接編集（Seed/運用手順に従う）して付与し、アプリ上の UI からは付与しない。

### 2.2 ロール別権限概要
- 一般ユーザー
  - 自アカウントのプロフィール閲覧（MyPage 上で表示）・編集
  - 自アカウントのパスワード変更（Identity 標準機能）
  - 自アカウントの削除（アカウント削除）
  - 自分のペットの登録・編集・削除（複数ペット対応）
  - 自分のペットの公開／非公開設定（IsPublic）変更
  - 自分のペットの健康ログ・スケジュール・通院履歴の CRUD
  - 自分の健康ログ／通院履歴への画像の登録・閲覧・削除
  - 他ユーザーの公開ペット（IsPublic = true）のプロフィール閲覧（ログイン済み同士のみ）

- 管理者ユーザー（Admin）
  - 管理者エリアへのアクセス
  - 全ユーザー一覧の閲覧
  - 任意ユーザーの削除（ユーザーおよび関連データの削除）
  - ※Admin であっても、他ユーザーの健康ログ／通院履歴／非公開ペット等を「閲覧できる」特権は付与しない（閲覧権限は通常の一般ユーザーと同等で、オーナーである場合のみ閲覧できる）。

## 3. 想定利用シナリオ（ユースケース）

### 3.1 一般ユーザー
- UC-01：ユーザー登録し、ログインする。  
- UC-02：MyPage 画面で、自分の表示名・メールアドレス・プロフィール画像と、自分のペット一覧を確認する。  
- UC-03：MyPage からプロフィール編集画面へ遷移し、表示名やプロフィール画像を更新する。  
- UC-04：MyPage からパスワード変更画面（Identity 標準 UI）へ遷移し、自分のパスワードを変更する。  
- UC-05：MyPage からアカウント削除画面へ遷移し、自身のアカウントと関連データを削除する。  
- UC-06：MyPage から新しいペットを登録する（初期状態では公開（IsPublic = true））。  
- UC-07：ペット登録後、公開したくない場合はペット編集画面で IsPublic を false（非公開）に変更する。  
- UC-08：MyPage で自分のペット一覧を確認し、各ペットカードからペット詳細画面へ遷移する。  
- UC-09：他ユーザーの公開ペット（IsPublic = true）一覧から、気になるペットのプロフィール詳細を閲覧する（ログイン済み同士のみ）。  
- UC-10：ペット詳細から、自分のペットの健康ログ／予定／通院履歴画面に遷移する。  
- UC-11：自分のペットの健康ログ（体重・食事量・活動量・排せつ状態・メモなど）を日々記録する。  
- UC-12：同じ日に朝と夜で別々に健康ログを記録する（日時で管理し、同一日の複数ログを許容）。  
- UC-13：健康ログに、当日の写真や食事の写真などを複数枚添付して記録する。  
- UC-14：通院後に診断内容や処方薬を通院履歴として記録し、診断書や検査結果などの画像を複数枚添付しておく。

### 3.2 管理者ユーザー
- UC-20：管理者としてログインし、ユーザー一覧で登録ユーザーの状況を確認する。
- UC-21：不要なテストユーザーや不適切なユーザーを削除する（関連するペット・健康ログ・予定・通院履歴・画像をまとめて削除）。

## 4. 機能要件

### 4.1 認証・ユーザー管理

**FR-001 ユーザー登録**  
- メールアドレスとパスワードによりユーザー登録できること。  
- 登録時に表示名（任意）を設定できること（未設定の場合はデフォルト表示名を用いる）。

**FR-002 ログイン／ログアウト**  
- 登録したメールアドレスとパスワードでログイン／ログアウトできること。

**FR-003 ユーザープロフィール編集**  
- ログインユーザーは、表示名とプロフィール画像を編集できること。

**FR-004 パスワード変更**  
- ログインユーザーは、Identity 標準機能でパスワード変更できること。

**FR-005 アカウント削除（ユーザー自身）**  
- ログインユーザーは、自分のアカウントを削除できること。  
- 削除時には、関連するデータ（ペット、健康ログ、予定、通院履歴、画像）を物理削除すること。  
- 画像ファイル削除に失敗した場合でも、DB レコードの削除は継続し、失敗したファイル識別子（StorageKey 等）を ILogger に出力すること。

**FR-006 未ログイン時のアクセス制御**  
- 未ログインユーザーは、アプリの機能画面（MyPage、ペット一覧・詳細、健康ログ等）にアクセスできないこと。  
- 未ログイン状態で保護された URL にアクセスした場合、ログイン画面にリダイレクトすること。  

---

### 4.2 MyPage 機能

**FR-010 MyPage 表示**  
- ログインユーザーは、自分専用の MyPage 画面を閲覧できること。  
- URL の一例：`/MyPage`。  
- MyPage には少なくとも以下を表示すること。
  - ユーザー表示名  
  - メールアドレス
  - プロフィール画像（またはデフォルト画像）
  - 自分のペット一覧

**FR-011 MyPage 上のペット一覧表示**  
- MyPage には、自分がオーナーであるペットのみの一覧を表示すること。  
- 各ペットについて以下の情報を表示すること。
  - ペット名  
  - ペット画像（サムネイル）  
  - 公開状態（公開／非公開）  

**FR-012 MyPage 内リンク**  
- MyPage から以下の画面に遷移できること。
  - プロフィール編集  
  - ペット登録  
  - ペット詳細  

**FR-013 ペット未登録時の表示**  
- ペット未登録の場合、MyPage に「ペットを登録してください」等の案内を表示すること。

---

### 4.3 ペット情報管理（プロフィール・公開設定）

**FR-020 全ペット一覧**  
- ログインユーザーは、公開ペット（IsPublic = true）の一覧を閲覧できること。  
- 一覧には少なくとも以下を表示すること。
  - ペット名  
  - 種別（Species）  
  - 品種（任意）  
  - オーナー表示名  
  - ペット画像（サムネイル）  
- 自分がオーナーのペットについては、IsPublic の値に関わらず一覧に表示してよい。  
- ソート：更新日時（または作成日時）の降順を基本とする。  
- ページング：10件/ページとする。

**FR-021 ペット詳細**  
- ペット詳細画面では、以下を閲覧できること。
  - ペット名  
  - 種別（Species：10択）  
  - 品種（任意）  
  - 性別（任意）  
  - 誕生日（任意）  
  - 迎えた日（任意）  
  - ペット画像（未設定時はデフォルト画像）  
  - オーナー表示名  
  - 公開状態（公開／非公開）  
- ペット詳細を閲覧できるのは、次のいずれかを満たす場合とする。
  - 対象ペットの IsPublic = true の場合  
  - ログインユーザーがそのペットのオーナーである場合  
- IsPublic = false のペットをオーナー以外が参照しようとした場合は 404 を返すこと（存在秘匿）。
- ペットのオーナーのみ、以下へのリンク（またはボタン）が表示されること。
  - 健康ログ一覧  
  - 予定一覧  
  - 通院履歴一覧  

**FR-022 ペット登録**  
- ログインユーザーは、ペットを登録できること。  
- 種別（Species）は 10択の選択式であること（自由記述は行わない）。  
- 新規登録時の公開設定 IsPublic の初期値は true（公開）とする（必要に応じて編集画面で非公開へ変更できる）。

**FR-023 ペット編集・削除**  
- ペットのオーナーは、ペット情報を編集・削除できること。  
- 他ユーザーのペット詳細画面には編集・削除機能を表示しないこと。

**FR-024 ペット検索・フィルタ**  
- ペット一覧画面で以下の条件で検索・絞り込みができること。
  - ペット名（部分一致）  
  - 種別（Species 10択／すべて）  
- 検索対象は IsPublic = true のペットおよび自分のペットとすること。

**FR-025 ペット公開設定変更**  
- ペットのオーナーは、ペット編集画面にて IsPublic（公開／非公開）を切り替えられること。  
- IsPublic = false のペットは、他ユーザーの一覧・検索結果・詳細画面からは見えないこと。  
- オーナー自身は IsPublic の値に関わらず、MyPage やペット詳細からアクセスできること。

---

### 4.4 健康ログ管理

**FR-030 健康ログ一覧**  
- ペットごとに健康ログの一覧を表示できること。  
- 表示項目例：  
  - 記録日時（RecordedAt）  
  - 体重  
  - 食事量  
  - 散歩時間／活動時間  
  - 排せつ状態  
  - メモ  
- 各健康ログに、画像が添付されている場合は「画像あり」であることが分かるように表示する。  
- ソート：RecordedAt 降順（同時刻の場合は Id 降順）。  
- ページング：10件/ページとする。

**FR-031 健康ログ登録**  
- ペットのオーナーは、そのペットの健康ログを登録できること。  
- 健康ログ登録時に、複数枚の画像を添付して保存できること（画像無し可）。  
- 同一日に複数件の健康ログを登録できること（日時で識別する）。

**FR-032 健康ログ編集・削除**  
- ペットのオーナーは、そのペットの健康ログを編集・削除できること。  
- 健康ログに添付された画像についても、個別に削除可能であること。  
- 編集時に、新たに画像を追加でアップロードできること。

**FR-033 健康ログ画像の閲覧**  
- ペットのオーナーは、健康ログに添付された画像を一覧・拡大表示できること。  
- 他ユーザーからは健康ログ自体が閲覧不可のため、画像も閲覧不可とする。

**FR-034 健康ログアクセス制御**  
- オーナー以外は他人のペットの健康ログを閲覧・登録・編集・削除できないこと。  
- 不正アクセス試行時は 403 を返すこと。

---

### 4.5 予防・スケジュール管理

**FR-040 予定一覧**  
- ペットごとに予定の一覧を表示できること。  
- 予定には少なくとも以下を保持できること：期日、種別、タイトル、メモ、完了フラグ。  
- 種別（Type）は固定値を推奨：`Vaccine` / `Medicine` / `Visit` / `Other`（表示名は UI 側で日本語化可）。  
- ソート：期日（DueDate）昇順を基本とする。  
- ページング：10件/ページとする。

**FR-041 予定登録**  
- ペットのオーナーは、そのペットの予定を登録できること。

**FR-042 予定編集・削除**  
- ペットのオーナーは、そのペットの予定を編集・削除できること。

**FR-043 完了フラグ更新**  
- ペットのオーナーは、予定の完了フラグ（完了／未完了）を切り替えられること。  

**FR-044 予定アクセス制御**  
- オーナー以外は他人のペットの予定を閲覧・登録・編集・削除できないこと。  
- 不正アクセス試行時は 403 を返すこと。

---

### 4.6 通院履歴管理

**FR-050 通院履歴一覧**  
- ペットごとに通院履歴の一覧を表示できること。  
- 表示項目例：通院日、病院名、診断、処方、メモ。  
- 各通院履歴に、画像が添付されている場合は「画像あり」であることが分かるように表示する。  
- ソート：通院日（VisitDate）降順。  
- ページング：10件/ページとする。

**FR-051 通院履歴登録**  
- ペットのオーナーは、そのペットの通院履歴を登録できること。  
- 通院履歴登録時に、複数枚の画像を添付して保存できること（画像無し可）。

**FR-052 通院履歴編集・削除**  
- ペットのオーナーは、そのペットの通院履歴を編集・削除できること。  
- 通院履歴に添付された画像についても、個別に削除可能であること。  
- 編集時に、新たに画像を追加でアップロードできること。

**FR-053 通院履歴画像の閲覧**  
- ペットのオーナーは、通院履歴に添付された画像を一覧・拡大表示できること。  
- 他ユーザーからは通院履歴自体が閲覧不可のため、画像も閲覧不可とする。

**FR-054 通院履歴アクセス制御**  
- オーナー以外は他人のペットの通院履歴を閲覧・登録・編集・削除できないこと。  
- 不正アクセス試行時は 403 を返すこと。

---

### 4.7 画像アップロード・配信・保存・デフォルト画像

**FR-060 画像アップロード（ユーザー）**  
- ユーザーはプロフィール画像をアップロードできること。  
- プロフィール画像未設定時は、あらかじめ用意したデフォルトプロフィール画像を表示すること。

**FR-061 画像アップロード（ペット）**  
- ペット登録・編集時に、ペット画像をアップロードできること。  
- ペット画像未設定時は、あらかじめ用意したペット用デフォルト画像を表示すること。

**FR-062 画像アップロード（健康ログ）**  
- 健康ログには、複数の画像ファイルを添付できること。  
- 1健康ログあたりの添付上限は最大10枚とする（既存画像＋追加画像の合算で判定する）。  

**FR-063 画像アップロード（通院履歴）**  
- 通院履歴には、複数の画像ファイルを添付できること。  
- 1通院履歴あたりの添付上限は最大10枚とする（既存画像＋追加画像の合算で判定する）。  

**FR-064 保存方法（ImageAsset）と参照方式**  
- アップロード画像は共通の画像エンティティ（ImageAsset）で管理し、アプリ側は **画像パスではなく ImageId / StorageKey を DB に保存**すること。  
- ApplicationUser / Pet は、各 1 枚の画像を `AvatarImageId` / `PhotoImageId` として参照すること（未設定可）。  
- HealthLog / Visit の複数画像は、関連テーブル（例：HealthLogImage / VisitImage）で `ImageId` を参照すること（表示順用に SortOrder を持てる）。  
- デフォルト画像は静的ファイルとして `wwwroot/images/default/` 配下に配置すること。

**FR-065 画像配信エンドポイント**  
- 画像配信は `GET /images/{id}` に統一し、**認可付きエンドポイント経由で返却**すること（デフォルト画像を除く）。  
- 認可（閲覧可否）は `ImageAsset.Category` と参照元（Pet/HealthLog/Visit/User）で決定すること。  
  - Avatar：**所有者のみ**  
  - PetPhoto：**(参照元 Pet が IsPublic = true) OR (所有者)** の場合のみ  
  - HealthLog：**所有者のみ**  
  - Visit：**所有者のみ**  
- 上記の参照元が解決できない（例：参照元レコード削除済み）場合は 404 を返すこと。  
- 挙動：
  - 未ログイン：ログイン画面へリダイレクト  
  - ログイン済みで認可 OK：画像返却  
  - ログイン済みで認可 NG：404（存在秘匿）  
- レスポンスヘッダ：
  - Cache-Control：`private, no-store`  
  - Content-Type：実ファイルに合わせる  
  - Content-Disposition：原則 `inline`（必要に応じて `attachment`）  

**FR-066 画像の安全加工（EXIF除去・向き正規化）**  
- アップロードされた画像は、保存前に Orientation を正規化し、再エンコードにより EXIF 等のメタデータを除去すること。  
- 対応形式は JPEG / PNG / WebP とし、HEIC/HEIF は非対応とする。  

**FR-067 画像容量・整合性（ユーザー上限・一時保存）**  
- ユーザーがアップロードした画像の合計容量は 100MB までとする（デフォルト画像は除外）。  
- 判定は「既存合計＋今回アップロード合計」でアップロード開始時に行い、超過する場合は **全体を失敗扱い**として保存しない。  
- 同時アップロードによる競合を避けるため、容量合計の更新・判定は DB トランザクションで整合性を担保すること。  
- ファイル保存と DB の整合性は以下で担保すること：  
  - 一時保存 → DB更新（トランザクション）→ 本保存（確定）  
  - 途中失敗時は一時ファイルを削除し、必要な補償処理（ImageAsset の削除／容量巻き戻し等）を行い、ILogger に出力すること。

### 4.8 管理者機能

**FR-070 管理者エリア**  
- Admin ロールを持つユーザーのみ、管理者エリアにアクセスできること。

**FR-071 ユーザー一覧（管理者）**  
- 管理者はユーザー一覧を閲覧できること。  
- 表示項目例：  
  - ユーザーID  
  - メールアドレス  
  - 表示名  
  - 登録日時  
  - ペット登録数  

**FR-072 ユーザー削除（管理者）**  
- 管理者は任意ユーザーを削除できること。  
- ユーザー削除時には、以下の関連データもまとめて削除すること（DB レコードの物理削除）。
  - ペット情報  
  - 健康ログ  
  - 健康ログ画像（関連テーブルおよび ImageAsset）  
  - 予定  
  - 通院履歴  
  - 通院履歴画像（関連テーブルおよび ImageAsset）  
- 画像ファイル削除に失敗した場合でも、DB レコードの削除は継続し、失敗したファイル識別子（StorageKey 等）を ILogger に出力すること。

**FR-073 管理者ロールの付与方法**  
- Admin ロールは DB 直接編集で付与する（UI からは付与しない）。  
- 初期投入は Seed 手順で行う。  
- 本番環境は Azure 上で管理者本人が実行し、開発環境はローカルで実行する。

## 6. 非機能要件

### 6.1 性能・規模
- 想定ユーザー数：数人〜数十ユーザー程度。  
- 同時アクセス数：少数（開発者本人＋閲覧者）。  
- 通常操作の応答時間：1〜2秒以内を目安とする。  
- 一覧画面は 10件/ページでページングし、必要に応じて DB 側でソート・フィルタを行う。

### 6.2 セキュリティ
- 全ページ HTTPS 通信を前提とする。  
- 認証は ASP.NET Core Identity による Cookie 認証とする。  
- パスワード・DB接続文字列・管理者メールアドレス等の機密情報はソースコードに直書きしないこと。  
  - 開発環境：`dotnet user-secrets` などで管理。  
  - 本番環境：Azure App Service の「構成」または Azure Key Vault で管理。  
- 画像アップロードにおいては少なくとも以下のバリデーションを行うこと。  
  - 許可する拡張子の制限（`.jpg`, `.jpeg`, `.png`, `.webp`）  
  - Content-Type の検証（`image/jpeg`, `image/png`, `image/webp`）  
  - ファイルサイズ上限（例：2MB 程度）  
  - 画像としてデコード可能であることの検証（拡張子偽装対策）  
- CSRF・XSS・SQLインジェクション等への対策は ASP.NET Core の標準機能を利用すること。  
- 画像は認可付きエンドポイントでのみ配信し、静的ファイルとして公開しない（デフォルト画像を除く）。

### 6.3 可用性・運用・画像保存
- 個人開発のため、厳密な可用性要件（SLA）は求めない。  
- 障害発生時は再デプロイ・再起動等により復旧できればよい。  
- アップロード画像は `wwwroot` 外の領域に保存する（例：App Service の永続ストレージ配下）。  
- 将来的に Azure Blob Storage など外部ストレージへ移行できるよう、画像保存・取得・削除処理はアプリケーション層のサービスクラスに抽象化する。  
- ユーザー／ペット／健康ログ／通院履歴の削除時には、DB レコードだけでなく対応する画像ファイルも削除対象とする。  
- 画像ファイル削除失敗時は ILogger に記録し、DB削除は継続する（孤児ファイルは必要に応じて運用で清掃する）。

## 7. データ要件（論理レベル）

### 7.1 主なエンティティ

- ユーザー（ApplicationUser）
  - Id（ASP.NET Identity 標準）
  - Email ほか Identity 標準項目
  - DisplayName（表示名）
  - AvatarImageId（ImageAsset.ImageId、任意）

- ペット（Pet）
  - Id
  - OwnerId（ユーザーID）
  - Name
  - SpeciesCode（後述の固定文字列）
  - Breed（任意）
  - Sex（任意）
  - BirthDate（任意）
  - AdoptedDate（任意）
  - IsPublic（公開設定）
  - PhotoImageId（ImageAsset.ImageId、任意）
  - CreatedAt / UpdatedAt（推奨）

- 健康ログ（HealthLog）
  - Id
  - PetId
  - RecordedAt（DateTimeOffset +09:00、必須）
  - Weight（任意）
  - MealAmount（任意）
  - ActivityMinutes（任意）
  - ExcretionStatus（任意）
  - Note（任意）

- 健康ログ画像（HealthLogImage）
  - Id
  - HealthLogId
  - ImageId（ImageAsset.ImageId）
  - SortOrder（表示順、任意）

- 予定（ScheduleItem）
  - Id
  - PetId
  - DueDate（必須）
  - IsDone（必須、完了フラグ。新規登録時は false）
  - Type（必須、最大 20 文字：Vaccine / Medicine / Visit / Other 等）
  - Title（必須、最大 100 文字）
  - Note（任意、最大 500 文字）

- 通院履歴（Visit）
  - Id
  - PetId
  - VisitDate（必須）
  - ClinicName（任意、最大 100 文字）
  - Diagnosis（任意、最大 500 文字）
  - Prescription（任意、最大 500 文字）
  - Note（任意、最大 1000 文字）

- 通院履歴画像（VisitImage）
  - Id
  - VisitId
  - ImageId（ImageAsset.ImageId）
  - SortOrder（表示順、任意）

- 画像（ImageAsset）
  - ImageId（GUID 等）
  - StorageKey（保存先キー：例 `images/{guid}.jpg`）
  - ContentType
  - SizeBytes
  - OwnerId（ユーザーID）
  - Category（Avatar / PetPhoto / HealthLog / Visit）
  - Status（Pending / Ready 等、推奨）
  - CreatedAt

### 7.2 Species（種別）コード一覧（固定）
- 表示名 → 内部値（固定文字列）
  - 犬 → `DOG`
  - 猫 → `CAT`
  - ハムスター・モルモット → `HAMSTER_GUINEA_PIG`
  - うさぎ → `RABBIT`
  - その他の哺乳類 → `OTHER_MAMMAL`
  - 小鳥 → `BIRD`
  - お魚 → `FISH`
  - 亀 → `TURTLE`
  - 爬虫類・両生類 → `REPTILE_AMPHIBIAN`
  - 昆虫 → `INSECT`

※表示名（日本語）は変更しても内部値は変更しない。

### 7.4 入力制約（必須／任意・値の範囲・文字数）

- 共通  
  - 特に指定のない文字列項目の最大長は 100 文字を目安とする。  
  - 日時は `DateTimeOffset(+09:00)` を基本とする（RecordedAt など）。

- ユーザー（ApplicationUser）
  - DisplayName：任意、最大 50 文字（未設定の場合はデフォルト表示名を用いる）

- ペット（Pet）
  - Name：必須、最大 50 文字  
  - SpeciesCode：必須（固定値。7.2 を参照）  
  - Breed：任意、最大 100 文字  
  - Sex：任意（「オス／メス／不明」などの固定値想定）、最大 10 文字  
  - BirthDate：任意  
  - AdoptedDate：任意  
  - IsPublic：必須（bool）、新規登録時は true を初期値とする  

- 健康ログ（HealthLog）
  - RecordedAt：必須  
  - Weight：任意（数値）  
  - MealAmount：任意（数値または短い文字列）  
  - ActivityMinutes：任意（整数分）  
  - ExcretionStatus：任意、最大 50 文字  
  - Note：任意、最大 1000 文字  

- 予定（ScheduleItem）
  - DueDate：必須  
  - IsDone：必須（bool）、新規登録時は false を初期値とする  
  - Type：必須、最大 20 文字（`Vaccine` / `Medicine` / `Visit` / `Other` など）  
  - Title：必須、最大 100 文字  
  - Note：任意、最大 500 文字  

- 通院履歴（Visit）
  - VisitDate：必須  
  - ClinicName：任意、最大 100 文字  
  - Diagnosis：任意、最大 500 文字  
  - Prescription：任意、最大 500 文字  
  - Note：任意、最大 1000 文字  

- 画像
  - 形式：JPEG / PNG / WebP のみ（HEIC/HEIF は非対応）  
  - サイズ：1ファイルあたり上限は 2MB 程度（詳細は基本設計で確定）  
  - 添付枚数：健康ログ・通院履歴は各 1 件あたり最大 10 枚
