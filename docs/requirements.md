- 文書名：ペット健康管理アプリ 要件定義書  
- バージョン：1.0 
- 作成日：2025-12-02  
- 作成者：Cat5Dog2
- 想定読者：開発者本人、レビュー担当者

---

## 1. システム概要

### 1.1 システム名

- ペット健康管理アプリ（仮称）

### 1.2 システム種別

- Webアプリケーション
  - バックエンド：.NET 10 (LTS) / ASP.NET Core MVC
  - 言語：C# 14（.NET 10 のデフォルト言語バージョン）
  - ORM：Entity Framework Core 10
  - 認証・認可：ASP.NET Core Identity

### 1.3 対象プラットフォーム・インフラ

- 開発環境
  - OS：Windows
  - DB：SQL Server LocalDB
  - 実行：ローカル（Kestrel）

- 本番環境（想定）
  - アプリホスト：Azure App Service
  - DB：Azure SQL Database
  - 通信：HTTPS（TLS）
  - 機密情報管理：App Service の「構成」（必要に応じて Azure Key Vault）

### 1.4 対象ユーザー・対象ペット

- 対象ユーザー
  - 一般のペットオーナー（一般ユーザー）
  - 開発者本人（管理者ユーザー）

- 対象ペット
  - 犬・猫を主な対象としつつ、「その他」の動物も登録可能とする。

### 1.5 システムの目的

- 一般ユーザーが複数のペットについて、以下の情報を一元管理できるようにする。
  - プロフィール情報（基本情報・画像）
  - 健康ログ（日々の体調・体重・食事・活動量など）
  - 予防・通院スケジュール
  - 通院履歴（診断内容・処方など）
- ペットのプロフィール情報は、公開設定（IsPublic）に応じて共有できるようにし、  
  デフォルトでは公開（IsPublic = true）として、必要に応じて「非公開」に切り替えられるようにする。
- ペットの詳細な健康情報（健康ログ・予定・通院履歴）は、登録オーナーのみ閲覧・編集可能とすることで、「プロフィールの共有」と「健康情報のプライバシー」を両立する。
- 健康ログ・通院履歴には複数の画像を添付できるようにし、診断書や検査結果などを視覚的に残せるようにする。
- 一般ユーザー自身がプロフィール更新やアカウント削除を行えるようにし、自己完結したアカウント管理を可能とする。
- 開発者本人のみが利用する管理者機能（ユーザー一覧・削除）を用意する。

## 2. 利用者とロール

### 2.1 ロール定義

- 一般ユーザー  
  メールアドレスとパスワードで登録・ログインして利用するペットオーナー。  
  自身のペットとその健康情報の管理主体。

- 管理者ユーザー  
  開発者本人を想定した特別ユーザー。  
  ASP.NET Core Identity のロール「Admin」を保持する。  
  「Admin」ロールは DB（主に本番 Azure SQL）を直接編集して付与し、アプリ上の UI からは付与しない。

### 2.2 ロール別権限概要

- 一般ユーザー
  - 自アカウントのプロフィール閲覧（MyPage 上で表示）・編集
  - 自アカウントのパスワード変更（Identity 標準機能）
  - 自アカウントの削除（アカウント削除）
  - 自分のペットの登録・編集・削除（複数ペット対応）
  - 自分のペットの公開／非公開設定（IsPublic）変更
  - 自分のペットの健康ログ・スケジュール・通院履歴の CRUD
  - 自分の健康ログ／通院履歴への画像の登録・閲覧・削除
  - 他ユーザーの公開ペット（IsPublic = true）のプロフィール閲覧

- 管理者ユーザー
  - 一般ユーザーの全権限
  - 全ユーザー一覧の閲覧
  - 任意ユーザーの削除（ユーザーおよび関連データの削除）

## 3. 想定利用シナリオ（ユースケース）

### 3.1 一般ユーザー

- UC-01：ユーザー登録し、ログインする。  
- UC-02：MyPage 画面で、自分の表示名・メールアドレス・プロフィール画像と、自分のペット一覧を確認する。  
- UC-03：MyPage からプロフィール編集画面へ遷移し、表示名やプロフィール画像を更新する。  
- UC-04：MyPage からパスワード変更画面（Identity 標準 UI）へ遷移し、自分のパスワードを変更する。  
- UC-05：MyPage からアカウント削除画面へ遷移し、自身のアカウントと関連データを削除する。  
- UC-06：MyPage から新しいペット（犬・猫など）を登録する（初期状態では公開（IsPublic = true））。  
- UC-07：ペット登録後、公開したくない場合はペット編集画面で IsPublic を false（非公開）に変更する。  
- UC-08：MyPage で自分のペット一覧を確認し、各ペットカードからペット詳細画面へ遷移する。  
- UC-09：他ユーザーの公開ペット一覧から、気になるペットのプロフィール詳細を閲覧する。  
- UC-10：ペット詳細から、自分のペットの健康ログ／予定／通院履歴画面に遷移する。  
- UC-11：自分のペットの健康ログ（体重・食事量・散歩時間・排せつ状態・メモなど）を日々記録する。  
- UC-12：同じ日に朝と夜で別々に健康ログを記録する（同一日付の複数ログ）。  
- UC-13：健康ログに、当日の写真や食事の写真などを複数枚添付して記録する。  
- UC-14：通院後に診断内容や処方薬を通院履歴として記録し、診断書や検査結果などの画像を複数枚添付しておく。  

### 3.2 管理者ユーザー

- UC-20：管理者としてログインし、ユーザー一覧で登録ユーザーの状況を確認する。  
- UC-21：不要なテストユーザーや不適切なユーザーを選択し、そのユーザーと関連するペット・健康ログ・予定・通院履歴・画像をまとめて削除する。

## 4. 機能要件

### 4.1 認証・ユーザー管理

**FR-001 ユーザー登録**  
- ユーザーはメールアドレスとパスワードを入力してアカウント登録できること。  
- ASP.NET Core Identity を利用すること。

**FR-002 ログイン／ログアウト**  
- 登録済みユーザーはメールアドレスとパスワードでログインできること。  
- ログアウト機能を提供すること。

**FR-003 ユーザープロフィール編集**  
- ログインユーザーは、プロフィール編集画面にて以下の項目を編集できること。
  - 表示名（DisplayName）  
  - プロフィール画像（アップロード）  
- プロフィール画像未設定の場合、デフォルトのプロフィール画像を表示すること。  
- プロフィール情報の「表示」は MyPage 上で行い、別途「プロフィール表示専用画面」は設けないこと。

**FR-004 パスワード変更**  
- ログインユーザーは、自分のパスワードを変更できること（旧パスワード認証＋新パスワード再入力）。  
- ASP.NET Core Identity の標準的なパスワードポリシーに従う。  
- MyPage からパスワード変更画面（Identity の管理画面）へのリンクを提供すること。

**FR-005 アカウント削除（ユーザー自身）**  
- ログインユーザーは、自分自身のアカウント削除操作を行えること。  
- アカウント削除実行前に確認画面（または確認ダイアログ）を表示し、誤操作を防ぐこと。  
- アカウント削除時には、以下を含む関連データがすべて削除されること（DB レコードの物理削除）。 
  - ユーザー（ApplicationUser）  
  - ユーザーがオーナーのペット（Pet）  
  - 各ペットの健康ログ（HealthLog）  
  - 健康ログ画像（HealthLogImage）  
  - 予定（ScheduleItem）  
  - 通院履歴（Visit）  
  - 通院履歴画像（VisitImage）  
- あわせて、対象ユーザーに紐づく以下の画像ファイルも削除対象とすること。
  - プロフィール画像ファイル  
  - ペット画像ファイル  
  - 各健康ログ画像ファイル  
  - 各通院履歴画像ファイル  
- アカウント削除完了後は、自動的にログアウト状態とし、同じ資格情報でログインできないこと。

**FR-006 未ログイン時のアクセス制御**  
- 未ログインユーザーは、MyPage やペット一覧・詳細などアプリの機能画面にアクセスできないこと。  
- 未ログイン状態で保護された URL にアクセスした場合、ログイン画面にリダイレクトすること。

### 4.2 MyPage 機能

**FR-010 MyPage 表示**  
- ログインユーザーは、自分専用の MyPage 画面を閲覧できること。  
- URL の一例：`/MyPage`。  
- MyPage には少なくとも以下を表示すること。
  - ユーザー表示名  
  - メールアドレス
  - プロフィール画像（またはデフォルト画像）
  - 自分のペット一覧

**FR-011 MyPage 上のペット一覧表示**  
- MyPage には、自分がオーナーであるペットのみの一覧を表示すること。  
- 各ペットについて以下の情報を表示すること。
  - ペット名  
  - 種別（犬／猫／その他）  
  - ペット画像（未設定時はペット用デフォルト画像）  
  - 公開状態（公開／非公開）のラベルまたはアイコン
- 各ペットカード（または行）には、ペット詳細画面（`/Pets/Details/{id}`）へのリンクを設けること。

**FR-012 MyPage 内リンク**  
- MyPage から次の画面へのリンクを提供すること：  
  - プロフィール編集画面（例：`/Account/EditProfile`）  
  - パスワード変更画面（例：`/Identity/Account/Manage/ChangePassword`）  
  - アカウント削除画面（例：`/Account/Delete`）  
  - ペット登録画面（例：`/Pets/Create`）

**FR-013 ペット未登録時の表示**  
- ユーザーに紐づくペットが1件もない場合、「登録されているペットはいません」等のメッセージと「ペットを登録」ボタンを表示すること。

---

### 4.3 ペット情報管理（プロフィール・公開設定）

**FR-020 全ペット一覧**  
- ログインユーザーは、公開ペット（IsPublic = true）の一覧を閲覧できること。  
- 一覧には少なくとも以下を表示すること。
  - ペット名  
  - 種別
  - 品種
  - オーナー表示名  
  - ペット画像（サムネイル）  
- 自分がオーナーのペットについては、IsPublic の値に関わらず一覧に表示してよい（フィルタで「自分のペットのみ」表示等も可）。

**FR-021 ペット詳細**  
- ペット詳細画面では、以下を閲覧できること。
  - ペット名  
  - 種別（犬・猫・その他を含む自由記述）  
  - 品種  
  - 性別  
  - 誕生日  
  - 迎えた日  
  - ペット画像（未設定時はデフォルト画像）  
  - オーナー表示名  
  - 公開状態（公開／非公開）  
- ペット詳細を閲覧できるのは、次のいずれかを満たす場合とする。
  - 対象ペットの IsPublic = true の場合  
  - ログインユーザーがそのペットのオーナーである場合  
- ペットのオーナーのみ、以下へのリンク（またはボタン）が表示されること。
  - 健康ログ一覧  
  - 予定一覧  
  - 通院履歴一覧  

**FR-022 ペット登録**  
- ログインユーザーは、自分のペットを新規登録できること。  
- 1ユーザーが複数ペットを登録可能とすること。  
- ペット画像を任意でアップロードして設定できること。  
- 新規登録時の公開設定 IsPublic の初期値は true（公開）とする。

**FR-023 ペット編集・削除**  
- ペット情報の編集・削除は、そのペットのオーナーだけが行えること。  
- 他ユーザーのペット詳細画面には編集・削除機能を表示しないこと。

**FR-024 ペット検索・フィルタ**  
- ペット一覧画面で以下の条件で検索・絞り込みができること。
  - ペット名（部分一致）  
  - 種別（犬／猫／その他／すべて）  
- 検索対象は IsPublic = true のペットおよび自分のペットとすること。

**FR-025 ペット公開設定変更**  
- ペットのオーナーは、ペット編集画面にて IsPublic（公開／非公開）を切り替えられること。  
- IsPublic = false のペットは、他ユーザーの一覧・検索結果・詳細画面からは見えないこと。  
- オーナー自身は IsPublic の値に関わらず、MyPage やペット詳細からアクセスできること。

---

### 4.4 健康ログ管理

**FR-030 健康ログ一覧**  
- ペットごとに健康ログの一覧を表示できること。  
- 表示項目例：  
  - 日付  
  - 体重  
  - 食事量  
  - 散歩時間／活動時間  
  - 排せつ状態  
  - メモ  
- 各健康ログに、画像が添付されている場合は「画像あり」であることが分かるように表示する。

**FR-031 健康ログ登録**  
- ペットのオーナーは、そのペットの健康ログを登録できること。  
- 健康ログ登録時に、複数枚の画像を添付して保存できること（画像無し可）。  
- 同一日付に複数件の健康ログを登録できること（朝・夜など時間帯別の記録を許容する）。

**FR-032 健康ログ編集・削除**  
- ペットのオーナーは、そのペットの健康ログを編集・削除できること。  
- 健康ログに添付された画像についても、個別に削除可能であること。  
- 編集時に、新たに画像を追加でアップロードできること。

**FR-033 健康ログ画像の閲覧**  
- ペットのオーナーは、健康ログに添付された画像を一覧・拡大表示できること。  
- 他ユーザーからは健康ログ自体が閲覧不可のため、画像も閲覧不可とする。

**FR-034 健康ログアクセス制御**  
- オーナー以外は他人のペットの健康ログを閲覧・登録・編集・削除できないこと。  
- 不正アクセス試行時は 403 を返すこと。

---

### 4.5 予防・スケジュール管理

**FR-040 予定一覧**  
- ペットごとに、予防・通院等の予定一覧を表示できること。  
- 表示項目：
  - 予定日  
  - 種別（ワクチン／薬／通院／その他）  
  - タイトル  
  - 完了フラグ  

**FR-041 予定登録**  
- ペットのオーナーは、そのペットの予定を登録できること。

**FR-042 予定編集・削除**  
- ペットのオーナーは、そのペットの予定を編集・削除できること。

**FR-043 完了フラグ更新**  
- 予定実施後、オーナーは完了フラグを更新できること。

**FR-044 予定アクセス制御**  
- 他人のペットの予定情報は閲覧・編集・削除できないこと。

---

### 4.6 通院履歴管理

**FR-050 通院履歴一覧**  
- ペットごとに通院履歴一覧を表示できること。  
- 表示項目：  
  - 通院日  
  - 病院名  
  - 診断内容  
  - 処方薬  
  - メモ  
- 各通院履歴に画像が添付されている場合は「画像あり」であることが分かるように表示する。

**FR-051 通院履歴登録**  
- ペットのオーナーは、そのペットの通院履歴を登録できること。  
- 通院履歴登録時に、複数枚の画像（診断書・検査結果など）を添付して保存できること（画像無し可）。

**FR-052 通院履歴編集・削除**  
- ペットのオーナーは、そのペットの通院履歴を編集・削除できること。  
- 通院履歴に添付された画像についても、個別に削除可能であること。  
- 編集時に、新たに画像を追加でアップロードできること。

**FR-053 通院履歴画像の閲覧**  
- ペットのオーナーは、通院履歴に添付された画像を一覧・拡大表示できること。  
- 他ユーザーからは通院履歴自体が閲覧不可のため、画像も閲覧不可とする。

**FR-054 通院履歴アクセス制御**  
- 他人のペットの通院履歴は閲覧・編集・削除できないこと。

---

### 4.7 画像アップロード・保存・デフォルト画像

**FR-060 画像アップロード（ユーザー）**  
- ユーザーはプロフィール画像をアップロードできること。  
- プロフィール画像未設定時は、あらかじめ用意したデフォルトプロフィール画像を表示すること。

**FR-061 画像アップロード（ペット）**  
- ペット登録・編集時に、ペット画像をアップロードできること。  
- ペット画像未設定時は、あらかじめ用意したペット用デフォルト画像を表示すること。

**FR-062 画像アップロード（健康ログ）**  
- 健康ログには、複数の画像ファイルを添付できること。  
- 画像は健康ログに紐づく形で保存・管理されること。

**FR-063 画像アップロード（通院履歴）**  
- 通院履歴には、複数の画像ファイルを添付できること。  
- 画像は通院履歴に紐づく形で保存・管理されること。

**FR-064 保存場所とパス表現**  
- アップロードされた画像ファイルは、`wwwroot` 配下の以下ディレクトリに保存すること。  
  - ユーザープロフィール画像：`wwwroot/upload/images/profile/`  
  - ペット画像：`wwwroot/upload/images/pet/`  
  - 健康ログ画像：`wwwroot/upload/images/healthlog/`  
  - 通院履歴画像：`wwwroot/upload/images/visit/`  
- 画像はアプリケーションから相対パス（例：`/upload/images/healthlog/xxxxx.jpg`）で参照すること。  
- デフォルト画像は以下に配置されたファイルを利用すること（ファイル名は基本設計で詳細定義）。 
  - プロフィール用：`wwwroot/images/default/profile.png`  
  - ペット用：`wwwroot/images/default/pet.png`

### 4.8 管理者機能

**FR-070 管理者エリア**  
- `/Admin/*` 配下に管理者専用エリアを設けること。  
- 「Admin」ロールを持つユーザーのみがアクセスできること。

**FR-071 ユーザー一覧（管理者）**  
- 管理者は全ユーザーの一覧を閲覧できること。  
- 表示項目：  
  - ユーザーID  
  - メールアドレス  
  - 表示名  
  - 登録日時  
  - ペット登録数  

**FR-072 ユーザー削除（管理者）**  
- 管理者は任意ユーザーを削除できること。  
- ユーザー削除時には、以下の関連データもまとめて削除すること（DB レコードの物理削除）。
  - ペット情報  
  - 健康ログ  
  - 健康ログ画像  
  - 予定  
  - 通院履歴  
  - 通院履歴画像  
- あわせて、対象ユーザーに紐づく以下の画像ファイルも削除対象とすること。
  - プロフィール画像ファイル  
  - ペット画像ファイル  
  - 各健康ログ画像ファイル  
  - 各通院履歴画像ファイル  
- 削除前に確認画面で対象ユーザーの概要および関連データ件数を表示すること。

**FR-073 管理者ロールの付与方法**  
- 「Admin」ロールの付与は、DB（`AspNetUsers` / `AspNetRoles` / `AspNetUserRoles`）を直接編集することでのみ行うこと。  
- アプリケーションの UI からロール付与・変更を行う機能は提供しないこと。

### 4.9 Identity 拡張機能の扱い

**FR-080 Identity 拡張機能のスコープ**  

- ASP.NET Core Identity が提供する以下の機能については、本バージョンではスコープ外とする。
  - パスワードリセット（メールによる再設定）  
  - メールアドレス確認（Email Confirmation）  
  - アカウントロックアウト（一定回数のログイン失敗によるロック）  
- 今後の拡張として検討する。

## 6. 非機能要件

### 6.1 性能・規模

- 想定ユーザー数：数人〜数十ユーザー程度。  
- 同時アクセス数：少数（開発者本人＋閲覧者）。  
- 通常操作の応答時間：1〜2秒以内を目安とする。

### 6.2 セキュリティ

- 全ページ HTTPS 通信を前提とする。  
- 認証は ASP.NET Core Identity による Cookie 認証とする。  
- パスワード・DB接続文字列・管理者メールアドレス等の機密情報はソースコードに直書きしないこと。  
  - 開発環境：`dotnet user-secrets` などで管理。  
  - 本番環境：Azure App Service の「構成」または Azure Key Vault で管理。  
- 画像アップロードにおいては少なくとも以下のバリデーションを行うこと（詳細は基本設計書で定義）。  
  - 許可する拡張子の制限（例：`.jpg`, `.jpeg`, `.png` のみ）  
  - Content-Type の検証（`image/jpeg`, `image/png` など）  
  - ファイルサイズ上限（例：2MB 程度）  
- CSRF・XSS・SQLインジェクション等への対策は ASP.NET Core の標準機能を利用すること。

### 6.3 可用性・運用・画像保存

- 個人開発・ポートフォリオ用途のため、厳密な可用性要件（SLA）は求めない。  
- 障害発生時は再デプロイ・再起動等により復旧できればよい。  
- 画像ファイルは当面、App Service のファイルシステム上（`wwwroot/upload/...`）に保存する前提とし、消失した場合は必要に応じて再アップロードで対応可能な範囲とする。  
- 将来的に Azure Blob Storage など外部ストレージへ移行できるよう、画像保存・取得・削除処理はアプリケーション層のサービスクラスに抽象化する。  
- ユーザー／ペット／健康ログ／通院履歴の削除時には、DB レコードだけでなく対応する画像ファイルも削除対象とすること。  
- 管理者ロール付与や本番データの操作は、Azure Portal や管理ツールから行う。

## 7. データ要件（論理レベル）

### 7.1 主なエンティティ

- ユーザー（ApplicationUser）
  - Id（ASP.NET Identity 標準）
  - Email ほか Identity 標準項目
  - DisplayName（表示名）
  - AvatarPath（プロフィール画像パス：`/upload/images/profile/xxxxx.jpg`）

- ペット（Pet）
  - Id
  - OwnerId（ユーザーID）
  - Name
  - Species（犬／猫／その他を含む文字列）
  - Breed
  - Sex
  - BirthDate
  - AdoptedDate
  - PhotoPath（ペット画像パス：`/upload/images/pet/xxxxx.jpg`）
  - IsPublic（プロフィール公開フラグ。true: 公開 / false: 非公開）

- 健康ログ（HealthLog）
  - Id
  - PetId
  - Date
  - WeightKg
  - FoodAmountGram
  - WalkMinutes
  - StoolCondition
  - Note

- 健康ログ画像（HealthLogImage）
  - Id
  - HealthLogId
  - ImagePath（`/upload/images/healthlog/xxxxx.jpg`）
  - SortOrder（表示順、任意）

- 予定（ScheduleItem）
  - Id
  - PetId
  - DueDate
  - Type（Vaccine / Medicine / Visit / Other など）
  - Title
  - Note
  - IsDone

- 通院履歴（Visit）
  - Id
  - PetId
  - VisitDate
  - ClinicName
  - Diagnosis
  - Prescription
  - Note

- 通院履歴画像（VisitImage）
  - Id
  - VisitId
  - ImagePath（`/upload/images/visit/xxxxx.jpg`）
  - SortOrder（表示順、任意）

- 詳細なテーブル構造およびリレーションは、別途作成する `docs/er-diagram.md` にて定義する。

### 7.3 入力制約（必須／任意・値の範囲・文字数）

- 共通  
  特に指定のない文字列項目の最大長は 100 文字を目安とする。

- ユーザー（ApplicationUser）
  - DisplayName：必須、最大 50 文字  

- ペット（Pet）
  - Name：必須、最大 50 文字  
  - Species：必須、最大 50 文字  
  - Breed：任意、最大 100 文字  
  - Sex：任意（「オス／メス／不明」などの固定値想定）、最大 10 文字  
  - BirthDate：任意  
  - AdoptedDate：任意  
  - IsPublic：必須（bool）、新規登録時は true を初期値とする  

- 健康ログ（HealthLog）
  - Date：必須  
  - WeightKg：任意、0.0〜200.0 の範囲（kg）  
  - FoodAmountGram：任意、0〜5000 の範囲（g）  
  - WalkMinutes：任意、0〜1440 の範囲（分）  
  - StoolCondition：任意、最大 100 文字  
  - Note：任意、最大 1000 文字  

- 予定（ScheduleItem）
  - DueDate：必須  
  - Type：必須、最大 20 文字（Vaccine / Medicine / Visit / Other など）  
  - Title：必須、最大 100 文字  
  - Note：任意、最大 500 文字  

- 通院履歴（Visit）
  - VisitDate：必須  
  - ClinicName：任意、最大 100 文字  
  - Diagnosis：任意、最大 500 文字  
  - Prescription：任意、最大 500 文字  
  - Note：任意、最大 1000 文字
